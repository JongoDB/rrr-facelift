# 15 — Zoho Flow (Current Automations)

> **What's running today in Zoho Flow.** This doc inventories the orchestrations we're replacing with n8n in Phase 02. Captured 2026-05-08 by driving the live console via the Chrome extension.

## Intake Automation — high-level architecture

The flow is huge — 4 effectively-parallel pipelines, one per `service_type × booking_urgency` combination, each repeating the same customer/estimate/invoice/comment work with slightly different parameters. The structure:

```
Trigger: Zoho Forms "New Customer Intake"  (real-time webhook)
   │
   ▼
Decision: Service Type? ─── Mobile ┐
                                   ├── Geocode Customer Address
                                   ├── ExtractCustomerCoordinates (script)
                                   ├── Get Driving Distance
                                   ├── RoundUpDistance
                                   ├── CalculateServiceFee
                                   ├── Set rounded_miles_NU... (variable)
                                   ├── Decision: Distance == 10? (and similar mileage thresholds)
                                   ├── BuildBatchMileageLine... (script)
                                   ├── Set mileage_over_ten_... (variable)
                                   ├── Refresh2Zoho (script — re-mints token?)
                                   │
                              On-Site ┐
                                   │   (skips geocode/mileage)
                                   │
                          Default ─┘
                                   │
                                   ▼
                            Send Email to RRR Su... (Zoho Mail — internal alert)
                                   │
                                   ▼
                  Decision: Booking Urgency?
                     Routine / Emergency-After-Hours / Default
                                   │
                                   ▼
            For each urgency branch (parallel pipelines):
                  Fetch customer (Zoho Books)
                  Decision: Fetched customer ID null?
                       ├── (null) → Create Customer  – E… / R…
                       └── (exists) → continue
                  Enable portal access (Zoho Books)
                  Send email (customer welcome)
                  Create estimate (Zoho Books)
                  Create task (Zoho Projects)
                  Add comment to estimate (Zoho Books)  ← THE intake template
                  ─── for invoice variants ───
                  Create invoice (Zoho Books)
                  Add comment to invoice (Zoho Books)   ← THE intake template
                  GET Invoice (raw API call)
                  AppendMileageLineItems… (script)
                  PUT New Invoice (raw API call)
                  Fetch invoice
                  CalculateDynamicCo… (script)
                  PUT New Invoice (raw API call again)
                  Send email (customer)
                  Send invoice (Zoho Books built-in)
                  Create task
                  …
                  On Error ► Send Error Email (to RRR support)
```

The 4 columns I see at zoomed-out level correspond to:
- **Mobile / Routine**
- **Mobile / Emergency / After-Hours**
- **On-Site / Routine**
- **On-Site / Emergency / After-Hours** (or similar pairing — exact mapping not separately confirmed)

## Why this matters for Phase 02

1. **The intake template auto-generation is owned by THIS flow's "Add comment to estimate/invoice" steps.** When n8n takes over, it must produce the exact same template format (already documented in [planning/06](06-zoho-integration.md) and [planning/14](14-zoho-org-schema.md)) — owners and techs read these comments daily.
2. **Mileage logic is fully encoded here.** Geocode → driving distance → rounded miles → conditional fee tiers → mileage-line builder. Phase 02's `02-process-intake-form` must reproduce this; the rate constants in [shared/src/constants.ts](../packages/shared/src/constants.ts) match what's already implied (10-mi free radius, $2.70/mi over).
3. **Per-urgency × per-service-type branching = 4 near-identical pipelines.** The current flow has heavy duplication because Zoho Flow's branching makes shared sub-flows awkward. **n8n can collapse this to one parameterized branch** using sub-workflows or function nodes — much easier to maintain.
4. **`Refresh2Zoho` and the raw `GET Invoice` / `PUT New Invoice` steps reveal a token-refresh hack.** The flow drops down to direct `https://www.zohoapis.com/books/v3/invoices/{id}` calls because Zoho Flow's built-in Books actions don't support modifying line items after creation cleanly. Our n8n equivalent uses the API directly via the `@rrr/zoho-tools` client we're building — no hack needed.
5. **`Create task` (Zoho Projects)** is currently part of the pipeline. n8n equivalent must keep that integration unless owner wants to drop it.
6. **`Enable portal access`** is set on every new customer. Phase 02's contact-creation step must replicate.
7. **Email notifications** flow through Zoho Mail (uses RRR's Workspace mailbox). Phase 02 sends customer email via Resend (per planning/03), but RRR-internal "new intake" alerts can stay on Workspace via SMTP relay or move to Resend too — owner's call.
8. **No Twilio SMS in the live Intake flow.** Twilio is registered as a connector but the active canvas uses email only. Either SMS was deferred or it's done out-of-band. Phase 02's plan to SMS the owner on new intake is an *addition*, not a replacement.

## Appointment Automation — paused flow

```
Trigger: Zoho Mail — "Email matching search…" (real-time)
   │
   ▼
Decision: Email body / subject ?
   ├── Appointment Requested ─┐
   │                          ├── ParseCancellationEmail (script)
   │                          ├── Set task_name (variable)
   │                          ├── Fetch task in project (Zoho Projects)
   │                          ├── Add comment (Zoho Projects)
   │                          ├── Fetch invoice (Zoho Books)
   │                          ├── …
   │                          (mirror branch on the right)
   │                          
   ├── Appointment Canceled ──┘
   │
   └── Default
```

A second, parallel branch on the right of the canvas does the same shape — likely the cancellation path versus request path.

This flow is **paused**. Plausible reasons (to confirm with the owner):
- Email-subject matching was unreliable
- Appointment changes are now handled differently (Zoho Books Customer Portal? manual?)

**For Phase 02:** don't blindly recreate. Ask the owner whether email-driven appointment handling still has a use, or if customer-portal-driven appointments are the future. The intake-form flow already covers initial scheduling.

## Mapping to our n8n target (Phase 02)

| Zoho Flow node type | n8n equivalent |
|---------------------|----------------|
| Zoho Forms trigger | HTTP Request webhook node (`/webhook/intake`), the website POSTs the form here directly — no Zoho Forms in the new world |
| Zoho Books action (Fetch / Create / Update) | HTTP Request → `api.triple-r-rv.com/zoho/<op>` (Worker dispatches via `@rrr/zoho-tools`) |
| Zoho Mail action | Resend node (customer emails) or SMTP node (RRR-internal alerts) |
| Zoho Projects action | Defer / replace with Postgres `tasks` table or skip (owner decision) |
| Decision node | n8n IF / Switch nodes |
| Variable Set / Function | n8n Set / Function nodes (TS in Code node, or call out to Worker for non-trivial logic) |
| `Refresh2Zoho` hack | Not needed — `@rrr/zoho-tools` caches token automatically |
| `GET / PUT Invoice` raw calls | Worker tools — `add_lines_to_invoice`, etc. |
